# GH 项目规则

## 这个目录是干什么的

- 这里放 `GH` 这个项目自己的长期规则。
- 这里只写稳定规则，不写临时任务安排。

## 项目身份

- `GH` 是一个全新项目。
- `GH` 默认独立于 Stillfield 和其他旧项目。
- 旧项目只能作为参考来源，不能作为默认继承来源。

## 硬边界

- 当前真实工作目录只有 `GH` 项目根目录。
- 在 `GH` 工作时，只以当前项目和全局系统内的文件为有效事实来源；目录外内容默认无效，除非用户明确授权引用。
- 不要跳过项目文档边界直接开工。
- 不要默认继承旧项目的产品逻辑、页面结构、命名方式、功能链路、技术假设和工作路径。
- 不要把参考壳子、旧设计稿、历史代码、实验代码当成当前真实产品。
- 只有在用户明确点名、明确授权、明确复用对象时，旧内容才可以被选择性引入。
- 不要创建对旧项目目录、旧工作流或隐藏路径的默认依赖。
- 不要在未确认真实操作对象前，对活项目进行覆盖、替换、迁移或重构。

## 默认执行原则

- 优先从小而真实的产品闭环开始，不要先做大而空的结构。
- 优先保证当前真实产品稳定，再考虑扩展和抽象。
- 项目方向没有确认前，不要偷偷滑入实现假设。
- 遇到高风险改动时，先确认对象、边界和可回退状态，再执行。

## 文件分工边界

- 全局协作方式写入全局系统。
- 全局长期记忆写入全局系统。
- 当前任务、当前阶段和当前焦点写入执行计划目录。
- 新想法写入想法收口目录。
- 已确认的方向变化写入决策目录。

## 后续怎么长

- 先把核心规则放在这个 `README.md`。
- 只有当某一类规则明显变多时，再单独拆成子文件。
- 没有实际内容压力前，不预先创建空文件。
