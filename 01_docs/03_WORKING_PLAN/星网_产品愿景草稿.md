## 立场陈述

`[A延长线 / 原文]`

> 没有星网，这些问题能不能解决？
>
> 能，但现在大多只能靠手工补丁解决。用户可以手动整理文件、维护规则、清理旧记忆、反复补 prompt，也能勉强把问题压住。但这套方法碎、脆、重度依赖用户自己充当系统集成师。
>
> **所以星网解决的不是"有没有办法"，而是"谁来解决"。**
>
> 星网要做的，是把这些原本分散、手工、脆弱的补丁，收敛成默认存在的产品能力。

## 核心理念：蒸馏

`[A延长线 / 原文，需补一句澄清]`

> 星网只做一件事：帮用户把自己蒸馏给 Agent。
>
> 不是"训练 Agent"，不是"喂数据"，是**蒸馏**。蒸馏的意思是把杂质去掉，留下本质。用户把自己最核心的思维方式、工作习惯、决策逻辑提炼出来，持续注入 Agent。
>
> *（待补：这里不是训练模型，是组织你交给 Agent 的内容。）*

## 设计原则：奥卡姆剃刀 + 六度分离

`[A延长线 / 原文]`

> 若无必要不增实体。简洁是地基，逻辑依然完整保留。
>
> 六度分离在星网里不是世界观装饰，而是产品结构原则：
>
> - 不是节点越多越强，而是路径越短越强
> - 节点要少，但连接要准
> - 结构要轻，但可达性要高
> - 星网做的不是知识堆积，而是最短路径组织
>
> **星网不是在制造复杂网络，而是在遵循六度分离的前提下，把用户意图最快传到正确节点。**

---

## 防漂移：约束链

`[A延长线 顶级 / 原文]`

> Agent 有上下文窗口限制，超过限制早期内容被挤出，Agent 开始忘事，判断开始漂移。
>
> 现在最有效的防漂移方法是文件化续接——把状态写进文件，下次读文件重建上下文。文件约束 Agent，防止漂移。
>
> 但这里有一个更深的问题：**文件本身也会漂移。**
>
> 旧记忆残留、路标失效、父子文件找不到——文件系统自己在漂移。你用文件约束 Agent，但没有东西在约束文件。
>
> **星网是在文件之上再加一层锚点：文件约束 Agent，星网约束文件。**
>
> ```
> Agent 容易漂移
>     → 用文件约束 Agent
>         → 文件本身也会混乱漂移
>             → 用星网约束文件
> ```

## 与 Obsidian 的关系

`[B延长线 / 原文，方向需要重审]`

> Obsidian 是用户喂给自己看的，星网是 Agent 喂给用户看的。
>
> Obsidian 让人像机器一样整理知识，星网让机器像人一样理解知识。方向完全相反。
>
> 星网保留文件系统的全部逻辑能力：查询、组织、权限、版本控制。不是阉割版，是升级版。

*（备注：这一段方向其实和补集 #5"双向资产"冲突——双向资产说的是"人和 Agent 都读"，
而这里说的是"Agent 喂给用户看"。下一步要决定：保留这个对比并修正方向，还是直接删掉。）*

## 多宇宙架构

`[A延长线 / 原文，"宇宙"一词可降权]`

> 星网不是一张图，是多个独立空间。防止节点无限膨胀，每个空间保持干净简洁。
>
> | 空间 | 内容 |
> |---|---|
> | 工作空间 | 项目文件、代码、任务节点 |
> | 赛博日记空间 | 个人思维、习惯、记忆、日记 |
> | 工作流空间 | 自动化流程、模板 |
>
> 关注点分离：你这个人 vs 这个项目 vs 可复用模板，分开存。
> 平台的 memory 是一坨，星网把它拆成三层。

## 补集元素清单

`[补集]`

下列元素是平台（Anthropic / OpenAI / Cursor 等）结构上不会、不能、不想做的事。

### 1. 本地优先 / 数据在用户磁盘
- 部署起点：localhost 本地网页 → Electron 桌面应用
- 用户记忆物理上存在用户自己的磁盘里
- 平台一定是云端 SaaS，记忆数据存在它们服务器
- 这是数据主权的物理基础

### 2. 跨 Agent 中立
- 同时支持 Claude Code / Codex / Cursor / VS Code
- 一份记忆在多个 Agent 里通用
- Anthropic 不会做"让用户也能在 Codex 里用同一份记忆"，OpenAI 也不会
- 只有第三方有动机做"跨 Agent 共用记忆层"

### 3. 用户对读取的主动控制权
- 用户能看见 Agent 这次读了哪些节点
- 用户能选择哪些节点这次不被读取
- 反平台黑盒：平台的 memory 是黑盒，用户不知道读了什么、记了什么、会不会用错

### 4. 寄生本地文件树
- 文件即节点，节点即文件
- 点击节点等于打开文件，编辑节点等于编辑文件
- 寄生文件树形态，不破坏原有结构
- 平台不会动用户的本地文件系统——这是 OS 层的事，超出平台边界

### 5. 双向资产（人和 Agent 都读）
- 同一份内容，用户自己也能读，Agent 也能读
- 平台的 memory 只服务 Agent，不服务人
- 用户写给自己的笔记/日记/工作流，天然是"人也能读"的资产

### 6. 视觉品牌 / 卡冈图亚美学
- 黑色深空背景，节点悬浮，引力光带连线
- 神经放电连线动效（Agent 读取节点时实时发光）
- 平台做的是工程师工具，不会做品牌化的视觉语言

### 7. 面向非技术用户
- 妈妈可以用，老板可以用，完全不懂技术的人也可以用
- 平台的 Agent 工具瞄准的是开发者和专业知识工作者

## 当前问题清单（下一步要决策的）

聚合之后能看到几个待解决的张力，留给下一步排序删减时处理：

1. **"蒸馏"理念 vs 补集 #5 双向资产**——蒸馏强调"用户喂给 Agent"，双向资产强调"人和 Agent 都读"。这两个是同一件事的两面，还是有冲突？

2. **Obsidian 对比段 vs 双向资产**——前者说"Agent 喂给用户看"，后者说"两边都读"。方向不一致，要不要保留对比段？

3. **多宇宙架构 vs 补集 #2 跨 Agent 中立**——多宇宙是星网内部的空间分割，跨 Agent 是星网对外的兼容性。两个都对，但放在一起会不会让定位变模糊？

4. **A 档表达整体气质 vs 补集清单的尖锐立场**——A 档段落是哲学/认知层面的论证（蒸馏、防漂移、六度分离），补集清单是产品立场层面的承诺（本地、跨 Agent、控权）。前者解释"为什么"，后者解释"是什么"。下一步要决定：哪一层做骨架，另一层做血肉。
