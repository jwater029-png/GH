# 地基层 · 需求来源:创始人手搭跨工具记忆体系复盘

> 日期:2026-06-04
> 性质:创始人亲身当了一次"星网的第一用户"——手工搭了一套 Claude + Codex 跨工具记忆体系,过程费时费脑。本文把这条手工路拆成步骤,作为**星网地基层(跨工具同步)的功能需求来源**。
> 一句话:**星网地基层 = 把下面这六个手工动作自动化,且产物全程可见。**

---

## 背景:今天手工搭了什么

- 现状是三套互不相通的记忆:① Claude 自带 auto-memory(自动)② 手写 CLAUDE.md / AGENTS.md 规则 ③ starnet 偏好库。
- 决定把"真东西"放进 VS Code 可见的 `projects/_SYSTEM/`(单一真相源),隐藏目录(`.claude`/`.codex`)只放指针。
- 给 Claude 补 `~/.claude/CLAUDE.md` 指针、给 Codex 用 `~/.codex/AGENTS.md`,接到 `_SYSTEM`。
- 发现并消除风险:Codex 那条原本靠**硬链接**同步(会断),已改成**独立静态指针文件**;并标注 auto-memory 在体系外、星网管不到。

**核心金句(产品哲学):** 用文件树的可见性,替换掉隐藏目录的不可见性。这给用户省下极大的大脑带宽——用户只需认两点:**一个 agent(可替换的前端)+ 一个大脑(自己的、跨工具的、看得见的后端)。换工具不换大脑。**

## 手工流程 → 星网自动化需求(7 步对照)

| # | 手工步骤 | 手工痛点 | 星网该自动做什么 |
|---|---|---|---|
| ① | 摸清现状:有几套记忆、各在哪、谁读谁 | 翻遍隐藏目录逐个 cat 才分清;同名 MEMORY.md 易混;普通人根本不知道有几套 | **一键扫描全机**,列出有哪些 AI 工具、各自记忆/规则文件在哪、谁自动加载;把不可见的隐藏目录画成看得见的图 |
| ② | 定"单一真相源"放哪 | 得先懂"哪个目录工具会自动读"才能决策 | **默认给一个可见的真相源目录**(markdown),不用用户懂工具内部机制 |
| ③ | 给每个工具补"入口指针" | 得知道 Claude 读 CLAUDE.md、Codex 读 AGENTS.md,文件名/格式/位置各不同,手写每一个 | **自动按每个工具认的格式、往正确位置投放指针**;新装工具自动补上 = "换工具不换大脑" |

| ⑦ | 边界标注 + 沉淀 | 得手动记住边界,今天不记下次又踩 | **把边界和规则常驻可见**,不依赖人脑记 |

## 提炼

本质是六个动作:**发现碎片 → 立真相源 → 多工具投放 → 验证 → 排雷 → 加固。**
星网地基层就是把这六个动作自动化,且产物全程落在用户看得见、能改的 markdown 里(markdown 主权,绝不退回黑箱)。

## 落地三盆冷水(与 product-core 对齐)

1. **翻译是持续工程,非一次性**:各工具格式会变,这根"水管"要一直修 → 属工程权衡(可选择只支持 Top 工具、格式变了再跟)。
2. **auto-memory 等工具自带记忆是"第二个大脑"**:星网必须答"吸纳还是划界",否则"一个大脑"承诺被拆穿 → **这条是前提假设的命门,非单纯权衡**(解法见战略:把自带记忆重定义为"被吸纳的数据源")。
3. **"自动"与"可见"天生打架**:必须守住可见这边(markdown 主权),否则等于把砍掉的黑箱偷偷捡回来 → 属产品承诺(已选可见优先)。

## 关联战略:破坏对手的前提假设

Claude auto-memory / Codex 自动记忆的隐含前提:**记忆属于工具、半黑箱、服务于单一工具的粘性。**
星网建立的对立前提:**记忆属于用户、可见可移植(markdown)、为跨工具自由流动而存在。**
推论:**对手的自动记忆越强,记忆碎片化越严重,星网这个"聚合+主权层"越有价值。** 它们不是竞品,是星网要吸纳的数据源。星网不在"谁的自动记忆更聪明"这条赛道里竞争,而是直接否定"记忆该归工具"这个前提。
