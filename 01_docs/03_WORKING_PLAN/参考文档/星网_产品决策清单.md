## 决策一:偏好注入策略(混合分层)

**问题:** 偏好怎么喂给 Agent?全塞进去费 token、稀释;全等 Agent 来查又靠不住。

**定论:混合分层。**

- **主动注入(保底层):** 每次会话开头,星网自动注入 ≤5 条"最高优先级 + 当前任务相关"的偏好。保证关键偏好一定生效,不靠 Agent 自觉。
- **按需查询(深入层):** 其余偏好通过 MCP 工具 `starnet.recall("关键词")` 暴露,Agent 想深入某方面时自己调取。

**为什么:** 关键偏好不能漏(否则主柱失效),但又不能每次灌几百条爆 token。≤5 条是经验上限,后续按种子用户反馈调。

**对应 plan:** 这就是护城河①**活注入**的落地机制(plan 第七节)——按当前任务挑相关的几条,不是全量灌;"省 token / 减少漂移"是它做对后的副产品,不单独立项。MCP server 见执行计划批次 1-②。

## 决策二:跨 Agent 偏好冲突处理(标记不仲裁)

**问题:** 两个 Agent 留了矛盾偏好(如 Claude 记 pytest、Cursor 记 unittest),星网怎么办?

**定论:星网不替用户仲裁,标记冲突,下次问用户。**

- 检测到矛盾 → 节点上打"冲突"标记,**不自动删任何一条**。
- **不打断当前工作流** — 不在 Agent 运行时弹窗。
- 用户下次打开星网桌面应用时,集中处理:留哪条 / 标成"分场景共存"。

**关键细节 — 先排除假冲突:** 很多"冲突"其实是分场景(A 项目用 pytest、B 项目用 unittest),不是真矛盾。判断真冲突前必须先比对 `applies-to` 字段(数据格式宪法已定义),**只有作用域重叠才算真冲突**,否则视为正常共存。

**为什么:** 偏好是用户的主权资产(产品哲学第 1、2 条),机器不该替用户做取舍。

**对应 plan:** 产品哲学"数据主权不退让 / 中立不退让";执行计划批次 1 活注入引擎一并处理冲突标记。

## 决策三:相关性激活规则(三层叠加打分)

**问题:** 怎么判断"当前任务"该激活哪些偏好?(写后端时不该弹前端偏好)

**定论:三层叠加打分,取得分最高的 ≤5 条注入(呼应决策一)。**

| 层 | 规则(2026-06-08 已对齐代码实际实现) | 可靠度 |
|---|---|---|
| 1. 优先级保底 + 全局钉子 | priority 基础分(high=3/med=2/low=1)任何语境都给;`always:true` 的偏好**永远注入、不被挤出**(如"用中文回复"/"用大白话") | 保底,不靠猜 |
| 2. 语境匹配 | 当前语境关键词命中偏好的 `context`(+4)/`tags`(+2)/`title`(+1)。信号两个来源:**SessionStart** 探项目语言(粗)、**UserPromptSubmit** 看你刚说那句话(细,2026-06-08 新增) | 看数据(标签)质量 |
| 3. 标签加分 | 已并入第 2 层(`tags` 命中 +2) | 细粒度补充 |

> ⚠️ **doc↔code 对齐(2026-06-08):** 本表原写"全局/项目路径/文件类型",那是旧语义(applies-to=项目路径)。**宪法 v3.0 把 applies-to 改成"哪些 agent"**,"项目路径/文件类型"在真实数据里都落到 `context`/`tags`,故代码合一为"语境匹配"。两个新机制(全局钉子 `always` + 看你说的话注入 UserPromptSubmit)已落地,见 `03_backend/mcp-server/`。
> **已知短板(批次2/3补):** 语境匹配是关键词子串比对——对标签好的偏好准,对"英文标签+中文提问"或措辞变体("写测试"vs"写个测试")会漏。鲁棒匹配 = 批次2 语义匹配(向量)+ 批次3 半自动蒸馏自动补中文标签。

- 各层打分,加总排序,取前 ≤5 条主动注入,其余留给 `recall()`。
- **手动 tag 不作主要依赖** — 用户画像就是不爱写偏好,再要求手动打 tag 不现实。tag 仅作可选加分项,有则用,无则不强求。

**为什么:** 这层是活注入(护城河①)的核心引擎 —— 精准激活而非全量加载,"省 token"只是顺带结果。

**对应 plan:** 第七节护城河①活注入;产品哲学"不学新东西、不被锁定"(所以不强制手动 tag)。

## 决策边界(明确不在阶段一做)

- **全自动从行为蒸馏偏好** → plan 推到**阶段四**。**注意:阶段一要做半自动蒸馏(观察→建议→一键确认,执行计划批次 3-①),系统提建议、用户拍板;阶段一不做的是"系统自己写偏好"的全自动。** 别把整个蒸馏都当阶段一不碰。
- **skill / 工作流资产** → **阶段三**(主柱的第二种资产,走偏好同一套管道)。
- **Cursor / Codex 适配** → **阶段二**。阶段一只验证 Claude Code 一条链路打通。

## 已知落地折扣(心里有数,不写进宣传)

**"所有 Agent 都能读"在非 Claude 工具上会打折:** 主动注入靠每个 Agent 给的注入点。Claude Code 有 hooks,能在会话开头动态塞 → 这条路通。Cursor 的 rules 是用户手动配的静态文件,星网只能生成/覆盖,做不到动态注入;Codex 能力又不同。

**含义:** 星网在 Claude Code 上能做出完整效果,在 Cursor/Codex 上分等级折扣。这是那些工具的能力边界,不是设计缺陷。对应 plan 风险表,宣传"你的偏好所有 AI 都能读"时落地要分等级——而且用好处句式,别说成"防漂移/替代"那类承诺(见 plan 第十一节)。
