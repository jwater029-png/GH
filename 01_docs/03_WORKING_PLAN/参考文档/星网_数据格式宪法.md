# 星网 数据格式宪法

> 📦 **星网文档集 v3** · 对齐于 2026/05/29 · 全套五份文档已同步
>
> **这是星网产品的数据格式标准。所有数据存储、读写、跨工具输出都必须遵守。**
>
> **宪法精神:字段只能追加,不能修改。** 这样老数据永远兼容,用户永远不需要"数据迁移"。

---

## 一、四条根本原则

1. **markdown + frontmatter 一切** — 所有用户数据都是 markdown 文件,frontmatter 存元信息,正文存内容。用户用任何文本编辑器都能看懂、能改、能迁移。

2. **字段名英文,内容可中文** — 字段名给程序读,内容给人和 AI 读。

3. **必填字段尽量少** — 用户写偏好不被字段卡住。绝大多数字段可选,有合理默认值。

4. **预留扩展字段** — 每个 frontmatter 都有 `extra: {}`,容纳未来字段,避免每次扩展都要数据迁移。

## 二、目录结构(整个 ~/.starnet/)

```
~/.starnet/
├── personal/                         # 个人空间(主柱核心)
│   ├── preferences/                  # 偏好
│   │   ├── pref-pytest-style-a3f2.md
│   │   └── pref-redis-rate-limit-b8c1.md
│   ├── habits/                       # 习惯
│   │   └── habit-functional-first-d4e2.md
│   ├── decisions/                    # 决策风格
│   │   └── decision-naming-style-e5f3.md
│   └── diary/                        # 赛博日记(阶段二上)
│       └── 2026-05-29.md             # 日记按日期命名,不用 id
│
├── projects/                         # 项目空间(灵魂层:节点图元数据)
│   └── <project-id>/                 # 每项目一目录,id = 项目根路径哈希
│       ├── meta.json                 # 项目元信息
│       ├── nodes.json                # 节点位置 + 连接(纯元数据)
│       ├── filters.json              # 隐藏规则(node_modules 等)
│       └── content/                  # 节点对应的 markdown 内容
│           ├── node-readme-a1b2.md
│           └── node-architecture-c3d4.md
│
├── workflows/                        # 工作流(阶段三上)
│   └── workflow-code-review-f6g7.md
│
├── adapters/                         # 跨 Agent 注入引擎自动生成,不手编
│   ├── claude/                       # 各项目的 .claude/CLAUDE.md
│   ├── cursor/                       # 各项目的 .cursor/rules
│   └── codex/                        # 各项目的 AGENTS.md
│
├── trash/                            # 删除节点 30 天后自动清理
│   └── 2026-05-29/
│
├── conflicts/                        # 同时编辑导致的冲突备份
│
├── index.db                          # sqlite + 向量索引(可重建,不是数据本体)
│
└── starnet.config.json               # 用户全局配置
```

**关键决策:**
- personal 下分子目录(preferences/habits/decisions/diary),不平铺。让 Agent 按类型加载,而不是每次都全读
- projects 用 project-id 不是项目名(用户改项目名 id 不变)
- adapters 是自动生成,不是手编。用户改 .cursor/rules 是无效的,要改星网偏好
- index.db 可重建。坏了从 markdown 重建,不丢数据

## 三、id 生成规则

**格式:** `<type-prefix>-<slug>-<short-uuid>`

**例子:**
- `pref-pytest-style-a3f2`
- `habit-functional-first-d4e2`
- `decision-naming-style-e5f3`
- `workflow-code-review-f6g7`
- `node-readme-a1b2`

**Type 前缀:**

| 类型 | 前缀 |
|---|---|
| preference | `pref` |
| habit | `habit` |
| decision-style | `decision` |
| diary | (不用 id,文件名是日期) |
| workflow | `workflow` |
| project node | `node` |

**Slug 部分:**
- 由用户写节点时输入的 title 自动生成
- 中文保留或转拼音(可配置),标点转横线
- 用户可以手动改 slug

**Short UUID 部分:**
- 4 位字符(36^4 = 1.6M 组合)
- 系统自动生成,用户不可改
- 防止同 slug 冲突

## 四、id ↔ 文件名双向锁死机制

**核心原则:用户改文件名和改 title 都行,但不能直接改 id。id 是系统管理的事。**

### 同步规则表

| 触发动作 | 自动同步行为 |
|---|---|
| 用户在外部把文件改名 `pref-pytest-style-a3f2.md` → `我的pytest偏好.md` | 星网检测 → 重写 frontmatter 的 id 为 `pref-我的pytest偏好-a3f2`(slug 跟着变,UUID 后缀保留) |
| 用户在星网里改 frontmatter 的 title | 星网把文件名同步改成新 slug + 原 UUID |
| 用户直接改 frontmatter 的 id | 星网拒绝(id 是系统不可改字段) |
| 用户改完后两边不一致 | 以**文件系统**为准(因为用户改名是更明确的意图) |

## 五、Frontmatter 字段宪法

### 通用核心字段(7 个)

所有 markdown 节点都有:

```yaml
---
id: pref-pytest-style-a3f2     # 系统管理,用户不可改
type: preference                # 类型,系统从所在子目录推断
title: 用 pytest 写测试         # 用户可改,改了 slug 跟着变
created: 2026-05-29
updated: 2026-05-29
applies-to: [claude-code, cursor, codex]   # 哪些 Agent 应该读
context: [testing, python]      # 在什么场景下激活(阶段四用)
priority: medium                # high / medium / low
tags: [test, python]            # 检索标签
extra: {}                       # 预留扩展字段
---
```

### 字段语义

**`id`** — 唯一标识,系统管理。引用稳定的根基。

**`type`** — 节点类型。值:`preference` / `habit` / `decision-style` / `diary` / `workflow` / `node`。

**`title`** — 显示名,用户可改。改完 slug 部分自动跟。

**`created` / `updated`** — ISO 日期。`updated` 每次修改自动刷新。

**`applies-to`** — 数组,哪些 Agent 应该读这条。值:`claude-code` / `cursor` / `codex` / `opencode` / `gemini-cli` 等。**默认值:** 全部 Agent(用户不填等于全适用)。

**`context`** — 数组,在什么场景下激活。阶段一不强用,字段预留。值由系统逐渐沉淀,如 `testing` / `coding-style` / `naming` / `error-handling` 等。

**`priority`** — `high` / `medium` / `low`。token 预算紧张时优先级决定谁先进。**默认值:** `medium`。

**`tags`** — 数组,自由标签。用户随便打。

**`extra`** — 字典,预留扩展。未来字段先放这里,稳定后再独立。

## 六、各类型节点示例

### 1. 偏好(personal/preferences/*.md)

```markdown
---
id: pref-pytest-style-a3f2
type: preference
title: 用 pytest 写测试
created: 2026-05-29
updated: 2026-05-29
applies-to: [claude-code, cursor, codex]
context: [testing, python]
priority: high
tags: [test, python, pytest]
extra: {}
---

我习惯用 pytest 写测试,不用 unittest。
风格上偏好函数式断言,少用 fixture 的复杂依赖注入。
```

### 2. 习惯(personal/habits/*.md)

```markdown
---
id: habit-functional-first-d4e2
type: habit
title: 函数式优先
created: 2026-05-29
updated: 2026-05-29
applies-to: [claude-code, cursor, codex]
context: [coding-style]
priority: medium
tags: [style, functional]
extra: {}
---

写代码先想能不能用纯函数解决,再考虑类和状态。
```

### 3. 决策风格(personal/decisions/*.md)

```markdown
---
id: decision-naming-style-e5f3
type: decision-style
title: 命名风格
created: 2026-05-29
updated: 2026-05-29
applies-to: [claude-code, cursor, codex]
context: [naming]
priority: medium
tags: [naming, style]
extra: {}
---

变量名用 snake_case,函数名 camelCase(JS)/snake_case(Py)。
讨厌缩写,宁可长不要短。
```

### 4. 赛博日记(personal/diary/*.md,阶段二上)

```markdown
---
id: diary-2026-05-29
type: diary
title: 2026-05-29 思考
created: 2026-05-29
updated: 2026-05-29
mood: focused                   # 可选

---

今天想清楚星网应该砍掉"看见 LLM 思考"这一块...
```

**注意:diary 的文件名 = 日期(`2026-05-29.md`),不是 id 格式。一天一文件。**

## 七、项目节点位置(nodes.json)

**不是 markdown,是 JSON。** 因为是程序生成的元数据,用户不直接编辑。

```json
{
  "version": 1,
  "project-id": "<project-id>",
  "nodes": [
    {
      "id": "node-login-a1b2",
      "file-path": "src/auth/login.ts",
      "x": 120,
      "y": 340,
      "user-tags": ["核心", "登录流程"],
      "connections": [
        {"to": "node-db-c3d4", "type": "depends-on"},
        {"to": "node-redis-e5f6", "type": "uses"}
      ],
      "last-accessed-by-agent": "2026-05-29T10:23:00Z"
    }
  ]
}
```

**节点的内容**(用户写的笔记/标签/说明)放在 `projects/<project-id>/content/<node-id>.md`,遵守通用 frontmatter 规则。

## 九、隐藏规则(filters.json)

```json
{
  "version": 1,
  "exclude": [
    "node_modules/**",
    ".git/**",
    "dist/**",
    "build/**",
    "__pycache__/**",
    ".next/**",
    "target/**",
    ".venv/**"
  ],
  "user-exclude": [],
  "user-include": []
}
```

**`exclude`** = 系统默认隐藏。
**`user-exclude`** = 用户额外隐藏。
**`user-include`** = 用户强制显示(覆盖 exclude)。

## 十、版本演进规则(宪法精神)

未来字段会变化,但要保证向后兼容:

**允许:**
- ✅ 加新字段(老文件没这个字段,程序读时给默认值)
- ✅ 字段加新枚举值(如 `applies-to` 加 `windsurf`)
- ✅ 修改默认值

**禁止:**
- ❌ 删字段(只能标记 deprecated)
- ❌ 改字段名(只能新增同义字段,逐步迁移)
- ❌ 改字段语义(同名字段语义变化 = 灾难)

---

## 十一、检查清单(写代码前必读)

读取节点时必查:

- [ ] frontmatter 格式合法(YAML 解析通过)
- [ ] id 字段存在且符合 `<type>-<slug>-<short-uuid>` 格式
- [ ] type 字段值在合法枚举内
- [ ] applies-to 是数组(空数组 = 默认全适用)
- [ ] 未知字段进 extra,不报错

写入节点时必做:

- [ ] 自动生成/更新 `updated` 字段
- [ ] 文件名 = id + .md(diary 除外)
- [ ] 字段顺序固定:id → type → title → created → updated → applies-to → context → priority → tags → extra → 其他
- [ ] 删除走 trash,不直接删
